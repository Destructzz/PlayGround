# Техническое задание (ТЗ): Интеграция Lounge и Event зон с Backend

В данном документе представлено подробное техническое задание (ТЗ) на реализацию полноценного функционала для Lounge-зон и Event-зон (событий). Документ описывает требования, бизнес-логику и интерфейсные параметры, ссылаясь исключительно на существующий код проекта.

---

## 1. Анализ существующей архитектуры и критические требования

### 1.1. База данных (Существующее состояние)
* В базе данных уже определен тип зоны [schema.sql:L4](file:///home/destruct/Projects/Web/PlayGround/backend/sql/schema.sql#L4):
  ```sql
  CREATE TYPE zone_type AS ENUM ('game', 'event', 'lounge', 'sys');
  ```
* Зоны хранятся в таблице `zones`, а бронирования — в таблице `bookings`.
* В существующей структуре при `PlaceID = NULL` (что естественно для лаунжей и событий) бэкенд выполняет жесткую проверку оверлапа.

### 1.2. Требование: Отключение эксклюзивной блокировки
В текущем файле сервиса бронирований [booking.go:L75-L89](file:///home/destruct/Projects/Web/PlayGround/backend/internal/service/booking.go#L75-L89) проверяется эксклюзивное пересечение:
```go
if dto.PlaceID == nil {
    overlapCount, err := b.queries.CheckZoneBookingOverlap(ctx, ...
```
* **Проблема:** Эта логика считает, что при `PlaceID = NULL` в зоне может находиться только одно бронирование. Это ломает логику Lounge (где несколько компаний могут сидеть параллельно в рамках емкости) и логику Event (где множество людей регистрируются на одно событие).
* **Требование:** Полностью исключить вызов `CheckZoneBookingOverlap` для зон с типом `'lounge'` и `'event'`. Вместо этого выполнять проверку суммарной свободной вместимости по количеству участников через существующий метод `CountActiveBookingsForZone`.

## 2. Требования к модернизации базы данных (Реляционный SQL-подход)

Мы отказываемся от использования универсального поля `details_json JSONB` для хранения основных настроек лаунжей и событий. Все сущности должны быть спроектированы в классическом реляционном стиле:

### 2.1. Сущность Событий (Event)
Необходимо спроектировать в миграциях полноценную структуру таблиц для сущности «Событие» (Event) и «Спикеры события». Мероприятия должны обладать следующими жесткими параметрами:
* Название события, описание, дата и время проведения (начало и окончание).
* Формат (например: турнир, митап, комьюнити-вечер).
* Вместимость участников.
* Цвет акцента для афиши фронтенда.
* Список приглашенных спикеров (связанная таблица).

### 2.2. Сущность Лаунжей (Lounge)
Свойства лаунж-зон должны храниться в связанных реляционных таблицах (связанных один-к-одному или один-ко-многим с таблицей `zones`):
* Настроение зоны (mood).
* Цвет акцента (подсветка).
* Премиум-статус (флаг premium).
* Список преимуществ зоны (perks, например: консоль, отдельный звук и т.д.) через отдельную таблицу связей.

### 2.3. Связь бронирований с событиями
Необходимо расширить общую таблицу `bookings`, добавив внешний ключ на таблицу событий. При регистрации пользователя на событие запись бронирования должна явно ссылаться на это событие.

## 3. Бизнес-логика и требования к бронированию

### 3.1. Фиксированная часовая сетка (Lounge)
* Бронирование лаунжа осуществляется строго на фиксированные интервалы времени кратно **1 часу** (например: 18:00 - 19:00, 19:00 - 20:00).
* Необходимо реализовать бэкенд-эндпоинт для проверки занятости лаунжа на конкретную дату:
  `GET /api/v1/public/lounge/{id}/availability?date=YYYY-MM-DD`
  Данный эндпоинт должен возвращать список фиксированных часовых слотов с указанием количества оставшихся свободных мест для каждого часа.

### 3.2. Автоматическое определение данных клиента
* На страницах бронирования лаунжей [lounge.vue](file:///home/destruct/Projects/Web/PlayGround/frontend/app/pages/lounge.vue) и событий [event.vue](file:///home/destruct/Projects/Web/PlayGround/frontend/app/pages/event.vue) необходимо **полностью удалить** все поля ввода контактных данных (имя гостя, телефон, Telegram, Email).
* Создание бронирования разрешено только авторизованным пользователям. Бэкенд должен автоматически подтягивать имя и контакты из активной сессии пользователя на основе токена авторизации.

## 4. Требования к интерфейсу пользователя (Frontend UX/UI)

### 4.1. Вкладки в стиле браузера (Sticky Tabs)
* Переключатели категорий, шагов бронирования или доступных дат должны быть оформлены в виде компактных горизонтальных вкладок, визуально напоминающих вкладки интернет-браузера.
* Вкладки должны быть «прилипшими» к верхней границе контейнера с информацией без больших отступов и спейсинга.

### 4.2. Скрытый блок информации о пользователе
* Информационный блок с профилем текущего авторизованного пользователя (аватар, имя, контактные данные, кнопка подтверждения), совершающего действие, должен быть убран с основного экрана.
* Данный блок должен вызываться отдельно через интерактивный элемент интерфейса (выезжающая панель-Drawer или модальное окно), так как эта информация не требуется перед глазами постоянно и занимает полезное пространство страницы.

## 5. Требования к реструктуризации панели администратора (Админка)

Для предотвращения разрастания файла [admin.vue](file:///home/destruct/Projects/Web/PlayGround/frontend/app/pages/admin.vue) (который сейчас перегружен различной логикой) устанавливаются следующие требования:

### 5.1. Разделение на независимые страницы
Админ-панель должна быть разделена на независимые подстраницы (Nuxt-маршруты) внутри новой папки `pages/admin/`:
* **Главный дашборд админа (`/admin`)** — общая статистика и лог действий.
* **Управление событиями (`/admin/event`)** — создание, удаление, редактирование спикеров событий и просмотр списков участников.
* **Управление лаунж-зонами (`/admin/lounge`)** — настройка настроения зон, цвета подсветки, списка perks, тарифов и вместимости.
* **Управление компьютерами (`/admin/pc`)** — настройки ПК-зон, мест и конфигураций.

### 5.2. Интерактивные формы управления
В формах добавления/редактирования зон и событий необходимо полностью исключить ручной ввод сырого JSON-текста. Все параметры (настройки преимуществ, даты, спикеры, цвета) должны редактироваться через специализированные интерактивные поля форм ввода.
