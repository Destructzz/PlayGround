# Логирование и Swagger/OpenAPI — стандарт проекта

Документ фиксирует:
- требования к логированию (структура, уровни, поля)
- правила корреляции запросов (request-id, trace-id)
- минимальный набор событий для диагностики
- подход к Swagger/OpenAPI (генерация, размещение, публикация)

---

## Логирование

### Принцип
- Логи — в первую очередь для **поиска причин** проблем и понимания поведения системы.
- В production предпочтительны **структурированные JSON логи**.
- Логи должны быть **коррелируемыми** по request-id.

### Уровни логирования (рекомендуемый минимум)
- `debug` — подробности, только для локалки/диагностики
- `info` — ключевые бизнес-события и факты успешной работы
- `warn` — подозрительные, но не критичные ситуации
- `error` — ошибки, которые требуют внимания (5xx, ошибки внешних систем)
- `fatal` — только на старте/инициализации, после этого процесс завершается

### Формат
- Local: допускается человекочитаемый текстовый формат
- Prod: JSON
- Формат задаётся через env (`APP_LOG_FORMAT`), уровень — через `APP_LOG_LEVEL`

### Обязательные поля в логах
- `ts` — timestamp
- `level` — уровень
- `msg` — сообщение
- `service` — имя сервиса
- `env` — окружение
- `request_id` — для HTTP запросов
- `method`, `path`, `status`, `duration_ms` — для request логов
- `user_id` / `account_id` (если есть) — только если это не PII и допустимо
- `err` — строковое представление ошибки (или отдельные поля), без утечек секретов

### Корреляция
- request-id генерируется на входе, пробрасывается во все логи в рамках запроса.
- Если request-id приходит извне (gateway), он должен быть принят и использован, иначе сгенерирован новый.

### Что логировать
- Старт/стоп сервиса (info)
- Готовность зависимостей (info/warn/error)
- Каждый HTTP запрос (info или debug в зависимости от политики)
- Ошибки инфраструктуры (db, внешние сервисы) — error
- Ошибки валидации входа — warn или info (по политике, но не error)
- Бизнес-события высокого уровня — info (без чувствительных данных)

### Что НЕ логировать
- пароли, токены, ключи, DSN целиком, содержимое секретов
- персональные данные без необходимости и согласованной политики
- большие payload тела запросов по умолчанию (включать только точечно и безопасно)

### Ошибки и stack traces
- В production stack trace логируется только для неожиданных ошибок (panic / internal).
- Для ожидаемых ошибок домена stack trace не обязателен.

## Swagger / OpenAPI

### Принцип
- Источник истины по контракту API должен быть единым.
- Контракт должен версионироваться вместе с кодом и меняться осознанно.

### Рекомендуемый подход
- Хранить спецификацию в репозитории в виде OpenAPI файла (yaml/json).
- Для документации публичного API держать `/api/v1` и соответствующий openapi для v1.
- В CI проверять:
  - валидность OpenAPI файла
  - совместимость (если у вас есть политика backward compatibility)

### Организация в репозитории
Рекомендуемые варианты:
- `api/openapi.yaml` как единый файл
- или `api/openapi/` если спецификация разбита по файлам и собирается в единый

Главное — чтобы итоговая спецификация была воспроизводима.

### Публикация Swagger UI
- Swagger UI должен быть доступен только в подходящих окружениях (обычно local/stage).
- В production — по политике безопасности (часто закрыт, или за auth).

### Версионирование и изменения
- Любое изменение эндпоинта должно сопровождаться изменением OpenAPI.
- Breaking changes должны приводить к новой версии API (например v2), если требуется сохранить совместимость.

### Документирование ошибок
- В OpenAPI должны быть описаны стандартные ответы ошибок:
  - 400 (validation)
  - 401/403 (auth)
  - 404 (not found)
  - 409 (conflict)
  - 429 (rate limit, если применимо)
  - 500 (internal)
- Модель ошибки должна быть единой и повторно используемой.

## Checklist перед merge (логирование + swagger)

- [ ] В логах нет секретов/PII
- [ ] У каждого HTTP запроса есть request-id и он попадает в логи
- [ ] Ошибки корректно классифицируются (warn vs error)
- [ ] OpenAPI обновлён для изменений API
- [ ] OpenAPI валиден и соответствует текущим маршрутам
