# Frontend

Разработка клиентской части системы осуществляется в соответствии с требованиями, описанными в документах `third_stage_requirements.md` и `sixth_stage_interfaces.md`.

На начальном этапе реализуется каркас пользовательского интерфейса без интеграции с серверной частью. Все данные на данном этапе используются в виде статических (hard-coded) значений, бизнес-логика и взаимодействие с backend отсутствуют.

## Общие принципы разработки
- единый визуальный стиль на всех страницах приложения;
- минимизация пользовательской и программной логики;
- приоритет простоты и читаемости кода;
- использование готовых компонентов и библиотек для сокращения объёма кода;
- адаптивная вёрстка для корректного отображения на различных устройствах;
- отсутствие дублирования стилей и компонентов.

## Технологический стек
- **Framework:** Nuxt 4 (Vue 3);
- **UI:** Tailwind CSS;
- **Подход к стилям:** использование утилитарных классов Tailwind CSS во всех случаях, где это возможно;
- **Компоненты:** переиспользуемые Vue-компоненты с минимальной логикой.

## Архитектура интерфейса
- каркас проекта формируется до реализации бизнес-логики;
- структура страниц и навигация полностью соответствуют спроектированным интерфейсам;
- все экраны реализуются в виде статических представлений;
- переходы между страницами реализуются средствами маршрутизации Nuxt.

## Адаптивность
- интерфейс разрабатывается по принципу mobile-first;
- поддержка основных разрешений экранов (mobile, tablet, desktop);
- использование responsive-классов Tailwind CSS;
- корректное масштабирование и перестроение элементов интерфейса.

## Технические особенности

Клиентская часть приложения реализуется с использованием фреймворка Nuxt, который предполагает файловую маршрутизацию. Все страницы приложения размещаются в каталоге `app/pages`, при этом структура файлов и папок напрямую определяет маршруты приложения.

Каждый файл в каталоге `pages` соответствует отдельной странице пользовательского интерфейса. Вложенные каталоги используются для формирования иерархии маршрутов. Данный подход упрощает навигацию и обеспечивает наглядное соответствие между структурой проекта и пользовательскими экранами.

Переиспользуемые элементы интерфейса (кнопки, формы, карточки, навигационные панели) выносятся в отдельные компоненты и размещаются в каталоге `components`. Это позволяет сократить дублирование кода и обеспечить единый стиль оформления.

Для стилизации интерфейса используется Tailwind CSS. Все стили задаются с помощью утилитарных классов, что исключает необходимость написания отдельных CSS-файлов в большинстве случаев. Адаптивность интерфейса достигается за счёт использования responsive-классов Tailwind.

На данном этапе разработки страницы не содержат бизнес-логики и не взаимодействуют с серверной частью. Данные отображаются в виде статических значений, что позволяет сосредоточиться на структуре интерфейса и пользовательском опыте.

Данный подход обеспечивает простоту структуры проекта, предсказуемость маршрутов и облегчает последующую интеграцию с серверной частью системы.
