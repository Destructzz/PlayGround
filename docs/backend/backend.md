# Go Backend (Gin) — дефолтный лэйаут и правила старта

Документ задаёт **стандарт проекта** для backend на Go с Gin: структура, правила модулей, конфиг, зависимости, стиль кода, обработка ошибок, тестирование и релизный подход.  
Код-примеров здесь **нет** (только договорённости и инструкции).

Смежные документы:
- `docs/backend/sqlc-postgres.md` — стандарт доступа к PostgreSQL через **sqlc + pgx**
- `docs/backend/logging-swagger.md` — стандарт логирования и Swagger/OpenAPI

## Цели

- Быстро стартовать проект с понятной структурой

## Рекомендуемый лэйаут репозитория

```
├── cmd/
│ └── api/ # точка входа HTTP сервиса
├── internal/
│ ├── app/ # сборка приложения (инициализация зависимостей)
│ ├── config/ # загрузка/валидация конфигурации
│ ├── http/
│ │ ├── server/ # роутинг, gin engine, middleware wiring
│ │ ├── handlers/ # HTTP handlers (контроллеры)
│ │ ├── middleware/ # middleware (request id, recover, auth, cors...)
│ │ ├── response/ # формат ответов + ошибки API
│ │ └── docs/ # если требуется хранить OpenAPI рядом (опционально)
│ ├── domain/ # доменные сущности, ошибки, интерфейсы
│ ├── service/ # бизнес-логика / use-cases
│ ├── repo/ # репозитории (DB доступ; см. sqlc-postgres.md)
│ └── observability/ # логирование/метрики/трейсинг (см. logging-swagger.md)
├── migrations/ # миграции БД (см. sqlc-postgres.md)
├── docs/
│ └── backend/
│ ├── README.md
│ ├── sqlc-postgres.md
│ └── logging-swagger.md
├── .env.example
├── .gitignore
├── go.mod
```

### Правила по папкам

- `cmd/*` — только точки входа. Никакой бизнес-логики.
- `internal/app` — “composition root”: единственное место, где собираются зависимости.
- `internal/http` — транспортный слой: маршруты, валидация входа, маппинг ошибок, ответы.
- `internal/service` — бизнес-правила. Не зависит от Gin/HTTP.
- `internal/repo` — доступ к данным. Зависимости от sqlc/pgx допустимы (см. `sqlc-postgres.md`).
- `internal/domain` — доменные типы, ошибки, интерфейсы. Минимум зависимостей.
- `pkg/` — использовать **только** если пакет реально нужен как внешний (для других модулей/реп). Иначе не добавлять.

## Конфигурация и переменные окружения

### Принцип
- В production конфиг **только через env vars**.
- Локально разрешено использовать `.env.local` (не коммитится), но .env — это **вспомогательный способ**.

### Файлы
- `.env.example` — коммитим, документируем.
- `.env.local` — личный файл разработчика, не коммитим.

### Именование env
- Префикс приложения: `APP_`
- Примеры: `APP_ENV`, `APP_HTTP_ADDR`, `APP_LOG_LEVEL`, `APP_DB_DSN`

### Валидация конфига
- Конфиг должен валидироваться на старте.
- При ошибке конфигурации сервис должен завершаться немедленно.

## HTTP слой (Gin) — стандарт

### Глобальные принципы
- Обработчики (`handlers`) содержат:
  - разбор/валидацию входа
  - вызов сервисов
  - маппинг ошибок домена/репозитория в HTTP статусы
  - формирование ответа по единому формату
- Обработчики НЕ содержат SQL и НЕ знают о структуре таблиц.
- `service` слой не импортирует Gin.

### Версионирование API
- Базовый префикс: `/api/v1`
- Поддержка `/healthz` и `/readyz` (или аналогов) — обязательна.

### Таймауты и лимиты
- Должны быть настроены:
  - таймауты чтения/записи/idle на HTTP сервере
  - ограничение размера body (там, где это имеет смысл)
- Любые внешние вызовы должны иметь контекст с дедлайном.

### Middleware (минимальный набор)
- request-id (генерация/проброс)
- recover/panic handler
- logging middleware (см. `logging-swagger.md`)
- CORS (если нужен)
- auth (если нужен)
- rate limiting (если нужен, на уровне API gateway или middleware)

## Единый формат ошибок и ответов

### Требования
- Все ответы API должны быть предсказуемыми.
- Ошибки должны:
  - иметь стабильный `code` (машиночитаемый)
  - включать человекочитаемое сообщение
  - опционально включать детали валидации
- В production нельзя “светить” внутренние stack traces или SQL детали в ответах.

### Маппинг ошибок
- Доменная ошибка → корректный HTTP status (4xx).
- Внутренняя ошибка/инфраструктура (DB недоступна и т.д.) → 5xx.
- Ошибки должны логироваться с достаточным контекстом (см. `logging-swagger.md`).

## Контексты и отмена запросов

- Каждый HTTP запрос должен иметь `context.Context`, который:
  - передаётся во все сервисы и репозитории
  - отменяет операции при разрыве соединения
- Запрещено:
  - создавать `context.Background()` внутри обработчиков/сервисов вместо передачи входного контекста
  - игнорировать отмену контекста в долгих операциях

## Тестирование

### Минимальный стандарт
- Unit-тесты на `service` слой (моки репозиториев через интерфейсы).
- Тесты на маппинг ошибок и валидацию входа (http слой).
- DB-интеграционные тесты допускаются, но должны быть изолированы и управляемы.

### Правила
- Любой новый бизнес-кейс должен сопровождаться тестом.
- Тесты должны быть детерминированными (без зависимостей на порядок выполнения).
- `-race` в CI обязателен.

## Качество кода и стиль

- `gofmt` обязательно.
- `golangci-lint` рекомендован, набор линтов фиксируется в репозитории.
- “Чистые” зависимости: минимизируем количество библиотек.
- Не тащим heavy frameworks, если есть простые стандартные решения.

---

## Dependecy injection и “сборка” приложения

- Вся сборка зависимостей происходит в `internal/app`.
- Слои зависят только “вниз”:
  - http → service → repo
  - domain не зависит от инфраструктуры
- Репозитории предоставляют интерфейсы для сервиса (или сервис принимает интерфейс репозитория).

---

## Релизный подход

- Версионирование API независимое от версий бинарника.
- Миграции БД выполняются явно отдельным шагом (см. `sqlc-postgres.md`).
- Любые критические изменения схемы — через миграции + обратимость (когда возможно).

## Хорошие практики (сводка)

### Делай
- держи бизнес-логику в `service`, HTTP — тонкий слой
- используй контекст везде
- делай единый формат ошибок
- проектируй репозитории вокруг use-cases, а не таблиц
- фиксируй практики в документации (как здесь)

### Не делай
- не смешивай SQL и HTTP handlers
- не лепи глобальные синглтоны без нужды
- не скрывай ошибки “тихо”, лучше маппинг + логирование
- не используйте `database/sql` для greenfield, если принято pgx+sqlc (см. `sqlc-postgres.md`)
