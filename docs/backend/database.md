# PostgreSQL: sqlc + pgx (greenfield стандарт)

Этот документ фиксирует стандарт доступа к PostgreSQL для нового Go backend:
- используем **sqlc** для генерации типобезопасного доступа
- используем **pgx v5** напрямую (без `database/sql`)
- используем **DSN** через env
- миграции — обязательная часть процесса разработки

---

## Принятое решение

- sqlc генерирует код с настройкой: `sql_package: "pgx/v5"`
- подключение к БД через `pgxpool`
- пакет `github.com/jackc/pgx/v5/stdlib` **не используется**
  - он нужен только для режима `database/sql`, который не применяется для нового проекта

---

## Структура файлов БД

Рекомендуется:

```
migrations/ # миграции схемы БД
internal/repo/db/ # место для сгенерированного sqlc кода (или internal/db)
internal/repo/ # репозитории, которые используют sqlc queries
sql/ # (опционально) исходники query.sql / schema.sql
```

### Рекомендация по размещению sqlc output
- держать output ближе к репозиториям, чтобы было понятно, что это часть data-access слоя
- автогенерированный код должен быть помечен как generated (sqlc делает это сам)

---

## DSN и конфигурация

- Источник истины: `APP_DB_DSN` (env var)
- DSN должен быть полным, включать:
  - пользователя/пароль
  - host/port
  - database
  - параметры подключения (например sslmode для local)

---

## Миграции схемы

### Общие правила
- Любое изменение схемы — только через миграцию.
- Миграции хранятся в репозитории.
- Порядок миграций должен быть очевидным (по времени/версии).

### Рекомендованный формат миграций
- “up/down” миграции (или иной стандарт), но выбранный формат фиксируется и не меняется без причины.
- Имена миграций должны быть говорящими:
  - `0001_init_schema`
  - `0002_add_users_table`
  - `0003_add_user_email_index`

### Процесс разработки
- Сначала пишем миграцию.
- Затем обновляем schema (если он хранится отдельно).
- Затем пишем/правим queries и гоняем sqlc.

---

## sqlc: организация запросов

### Правила на queries
- Каждый запрос должен иметь:
  - понятное имя
  - стабильную сигнатуру
  - предсказуемую семантику (не “магические” side effects)
- Избегаем “гигантских” запросов без нужды; лучше несколько простых, но не в ущерб производительности.

### Именование
- CRUD: `CreateX`, `GetXByID`, `ListX`, `UpdateX`, `DeleteX`
- для специфических use-case: `ListXByY`, `GetXForZ`, `LockXForUpdate` и т.п.

### Работа с транзакциями
- Транзакции применяются там, где требуются гарантии согласованности.
- Транзакции управляются на уровне `service` (use-case), репозитории/queries поддерживают работу через общий интерфейс (pgx Tx).

---

## Репозитории vs sqlc queries

### Рекомендуемый паттерн
- sqlc queries — “низкоуровневый доступ” к таблицам/запросам
- репозитории — “адаптер под домен/use-case”, в котором:
  - прячется структура таблиц
  - делается маппинг типов
  - обрабатываются ошибки и маппятся в доменные

### Запрещено
- тянуть sqlc queries напрямую в HTTP handlers
- возвращать “сырой” DB тип наружу без маппинга, если это ломает доменную модель

---

## Ошибки и их интерпретация

- Уникальные ограничения, not found, foreign key — должны маппиться в доменные ошибки.
- DB/infra ошибки — должны оборачиваться как internal и логироваться.
- Нельзя “светить” SQL/таблицы наружу в сообщениях клиенту.

---

## Производительность и безопасность

- Всегда использовать параметризованные запросы (sqlc делает это, если запрос написан корректно).
- Следить за индексами на часто используемых фильтрах.
- Пагинация обязательна там, где потенциально много строк.
- Избегать N+1 на уровне API; лучше один запрос с join/CTE, если это оправдано.

---

## Локальная разработка (без Docker для окружения)

- Поднимаем Postgres локально (native install).
- Создаём базу и пользователя.
- Прописываем `APP_DB_DSN` в `.env.local`.

> Конкретные команды установки отличаются по ОС/дистрибутиву и не фиксируются в стандарте проекта.
