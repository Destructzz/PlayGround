# AI Review Guide — правила проверки кода и архитектуры (Gin + sqlc + pgx)

Этот документ написан **по моей просьбе**: я хочу, чтобы нейронка могла **проверять мой код** и подсказывать правки **строго по архитектурным правилам проекта**.

## Обязательное правило для нейронки перед любыми правками

Перед тем как что-то менять/предлагать, нейронка **должна задать вопросы** и дождаться ответов, потому что без контекста легко испортить архитектуру.

Нейронка обязана спросить:

1) **Где именно я сейчас работаю?**  
   - путь до файла/папки, например: `internal/http/handlers/user.go`  
   - или ветка/задача

2) **Какой слой это должен быть?** (выбрать одно)
   - `http` (transport)
   - `service` (use-case)
   - `repo` (data-access)
   - `domain` (types/errors/interfaces)
   - `app` (composition root)

3) **Что является целью изменения?**  
   - новая фича / исправление бага / рефакторинг / оптимизация / тесты

4) **Есть ли ограничения по “не трогать”?**
   - нельзя менять публичные API
   - нельзя менять схему БД
   - нельзя менять формат ответов
   - нельзя менять роуты/контракт

Пока ответы не получены — **нейронка не должна переписывать код**, только задавать вопросы и показывать “план проверки”.

---

## Главная цель: архитектура по папкам (работаем там, где положено)

Проект строится по слоям, и **место кода важнее, чем удобство прямо сейчас**.

### Базовый лэйаут (истина проекта)

- `cmd/*` — только точка входа
- `internal/app` — сборка зависимостей (единственное место, где “склеиваем” всё)
- `internal/http/*` — Gin, роуты, middleware, handlers, формат ответов
- `internal/service/*` — бизнес-логика (use-cases)
- `internal/repo/*` — доступ к данным, sqlc/pgx
- `internal/domain/*` — доменные типы, доменные ошибки, интерфейсы (минимум зависимостей)

### Жёсткие правила зависимостей (dependency direction)

Разрешённые зависимости:

- `http -> service -> repo`
- `service -> domain`
- `repo -> domain`
- `app -> http/service/repo/config/observability`

Запрещённые зависимости:

- `service` не импортирует `gin` / `http`
- `domain` не импортирует `pgx`, `sqlc`, `gin`, `zap/zerolog` и т.п.
- `handlers` не делают SQL и не знают про таблицы/колонки
- `repo` не формирует HTTP-ответы и не знает про статусы

Если нейронка видит нарушение — она должна:
- указать место нарушения
- предложить перенос ответственности в правильный слой
- описать минимальный дифф-рефакторинг

## Стандарт: PostgreSQL доступ только через sqlc + pgx (greenfield)

### Принятое решение
- Используется `sqlc` с `sql_package: "pgx/v5"`
- Используется `pgxpool`
- `database/sql` и `github.com/jackc/pgx/v5/stdlib` **не используются**

### Проверка нейронкой
Нейронка должна подсветить как ошибку, если:
- в проекте появляется `database/sql` без явного разрешения
- подключается `pgx/v5/stdlib`
- handlers напрямую используют sqlc queries или pgx

---

## Стандарт: HTTP слой (Gin) — тонкий транспорт

### Что делает handler
- разбирает вход (path/query/body)
- валидирует
- вызывает `service`
- маппит ошибки в HTTP статусы
- формирует ответ в едином формате

### Что НЕ делает handler
- не пишет SQL
- не управляет транзакциями
- не содержит бизнес-правила (“если VIP, то…”) — это `service`

### Проверка нейронкой
Нейронка должна флагать:
- бизнес-логика внутри `internal/http/handlers`
- прямые обращения к БД из handler
- форматирование/логика ответа “вразнобой” (без единого response/error формата)

---

## Стандарт: service слой (use-cases)

### Роль service
- бизнес-правила
- оркестрация операций
- транзакционные границы (если нужны)
- доменные ошибки

### Вход/выход service
- принимает DTO/параметры уровня домена или “команду”
- возвращает доменные модели/DTO и доменные ошибки
- не возвращает HTTP коды и не знает про JSON

### Проверка нейронкой
Флагать:
- импорт `gin`, `net/http` в `internal/service`
- возврат “HTTP ошибок” вместо доменных
- логирование в стиле “всё подряд” без контекста

---

## Стандарт: repo слой (data-access)

### Роль repo
- инкапсулировать sqlc queries
- маппить ошибки БД в доменные
- не раскрывать наружу структуру таблиц там, где это ломает домен

### Ошибки
- уникальные ограничения / not found / fk — маппим в доменные ошибки
- остальные — internal

### Проверка нейронкой
Флагать:
- repo возвращает сырые ошибки pgx без маппинга
- repo знает про HTTP
- отсутствует пагинация в “list” запросах

---

## Стандарт: domain слой (самый чистый)

### Что внутри
- типы, сущности, value objects (если используете)
- доменные ошибки и коды
- интерфейсы портов (repo interfaces), если так принято

### Что запрещено
- внешние зависимости на веб/базу/логирование
- “утилитный мусор” без доменного смысла

---

## Контекст и отмена (обязательное)

- Вся цепочка: `handler -> service -> repo` должна принимать `context.Context`.
- Нейронка должна флагать:
  - создание `context.Background()` внутри запроса
  - потерю контекста при вызовах

---

## Единый формат ответов и ошибок (обязательное)

- API ответы должны быть единообразными.
- Ошибка должна иметь:
  - `code` (стабильный)
  - сообщение
  - детали валидации (если применимо)

Нейронка должна:
- проверять, что ошибки не “утекают” наружу (SQL, stacktrace, секреты)
- проверять, что статус-коды согласованы с кодами ошибок

---

## Тестируемость (архитектурное требование)

Нейронка должна проверять:
- сервисный слой тестируется через интерфейсы репозиториев
- бизнес-правила не спрятаны в handler
- критичные use-cases покрыты unit тестами

---

# Как нейронка должна проводить ревью (обязательный алгоритм)

1) Определить файл и папку (слой).
2) Проверить, соответствует ли код роли слоя.
3) Проверить направление зависимостей.
4) Проверить контекст, ошибки, формат ответов.
5) Проверить границы ответственности (handler/service/repo).
6) Дать вывод:
   - ✅ соответствует
   - ⚠️ частично (что именно)
   - ❌ нарушает (и как исправить минимально)

---

# Что ещё я бы добавил в инструкцию (рекомендации к расширению)

Нейронка должна уточнить, нужно ли добавить и зафиксировать в документации:

1) **Стандарт именования**
   - пакеты, файлы, методы, SQL запросы (Create/Get/List/Update/Delete)

2) **Политика транзакций**
   - кто открывает транзакцию (обычно service)
   - как репозитории принимают executor/tx

3) **Стандарт миграций**
   - формат up/down
   - правила именования и обратимости

4) **Логирование**
   - обязательные поля (request_id, duration_ms, status)
   - уровни логов
   - запрет на логирование секретов/PII

5) **Swagger/OpenAPI**
   - где хранится spec
   - политика версий `/api/v1`
   - правила для breaking changes

6) **Границы модулей**
   - если проект вырастет: разделение на модули (user/billing/etc.)
   - правило “модуль не импортирует внутренности другого модуля”

## Формат запроса к нейронке (шаблон)

Когда я прошу проверить код, я буду писать так:

- Путь: `<path>`
- Цель: `<feature/fix/refactor>`
- Слой: `<http/service/repo/domain/app>` (если знаю)
- Ограничения: `<не менять X>`
- Контекст: `<что должно работать>`

или не буду говорить, по типу ```проверь что нет так```

Нейронка обязана начать с вопросов из раздела “Обязательное правило”, если чего-то не хватает.

## Контролируемые исключения из правил (не считать архитектурной ошибкой)

В проекте допускаются **локальные исключения**, которые не являются поводом “перестраивать архитектуру” и не считаются нарушением слоёв, если они:

- **малые по объёму** (обычно 1 файл / 1 пакет),
- **изолированы** и не тянут за собой зависимости “вверх по слоям”,
- **не содержат бизнес-логики**,
- не становятся “шаблоном” для остального кода.

### Разрешённые исключения (примерные категории)

1) **HTTP util endpoints**
- `healthz`, `readyz`, `metrics`, `version`, `build-info`
- Они могут жить рядом с HTTP сервером и использовать простые проверки.

2) **Response helpers**
- Пакет `internal/http/response` может содержать:
  - структуры ответа
  - сериализацию ошибок
  - маппинг доменных ошибок в API формат
- Это не считается “логикой домена”, это инфраструктурный слой API.

3) **Простые dependency checks**
- Для readiness допускаются прямые “пинги” зависимостей (например DB ping),
  если:
  - это не превращается в общий репозиторий/бизнес-логику
  - и не протаскивает sqlc queries в handlers

### Жёсткие ограничения на исключения (чтобы не было разрухи)

Исключения НЕ дают право:
- писать бизнес-правила в handlers
- делать SQL запросы из handlers, кроме простого ping/check
- тянуть `repo/service` зависимости в `domain`
- добавлять “одиночные хелперы” в случайные папки (утилиты должны иметь дом)

### Правило для ревью-нейронки

Если обнаружен код из списка исключений, нейронка должна:
- НЕ предлагать перенос “ради чистоты”
- проверить только изоляцию и отсутствие бизнес-логики
- предложить улучшения в пределах того же слоя (например, единый формат ответа, корректные статусы, логирование)

Если исключение начинает разрастаться — это сигнал вынести его в отдельный пакет внутри соответствующего слоя
(например `internal/http/health`, `internal/http/response`), а не ломать весь лэйаут.
